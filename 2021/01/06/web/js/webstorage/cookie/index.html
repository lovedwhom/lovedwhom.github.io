cookie与session

cookie

客户端请求服务器时,如果服务端需要记录用户状态,服务端就通过response Header向客户端颁发cookie,
而客户端浏览器会把cookie保存起来,当浏览器再次请求服务器时,浏览器会把请求地址连同该cookie一同
提交给服务器,服务器通过检查该cookie来获取用户状态

session

当服务器接收到请求时,就从存储在服务器上的无数session信息中查找客户端请求时带过来的cookie的状态,
如果服务器中没有这条session则添加一天session信息

通常cookie中存的是session信息经过计算后的唯一Id.

1. cookie存储在哪里

cookie一般是被浏览器以txt的形式存储在电脑硬盘中,供该浏览器进行读\写操作

2.cookie是如何工作的

request

当浏览器发送请求时,浏览器会自动检查是否有相应的cookie,如果有则将cookie添加到Request Header的cookie字段中

response

当服务端需要种cookie时,在http请求中的Response Header中添加Set-Cookie字段,浏览器接收到之后会自动解析识别,将cookie种下

3.cookie和session的区别

  • 存储位置不同: cookie数据存储在浏览器上,session存储在服务器上
  • 存储大小不同: 一般单个cookie保存的数据不能超过4k,单个域名最多保存30个cookie(不同浏览器有差异)
    session则无大小和数量限制

4.什么时session级别的cookie

session级别的cookie只针对当前会话存在,会话终止则cookie消失

当cookie没有设置expires的时候,该cookie只会在会话期间存在,当浏览器退出的时候,该cookie就会消失.
浏览器的表现味Expires/Max-Age 的内容为Session

5.cookie可以被谁来操作

服务端和js抖可以读/写cookie

6.cookie各种属性详情

Name: cookie名
Value: cookie值
Domain: cookie的域名. 如果设成[.example.com] 那么子域名 a.example.com 和 b.example.com 都可以使用.example的cookie 反之则不可以
Path: 允许读取cookie的url路径 一般设置为/ 访问子路径时,会包含其父路径的coolie 而在访问父路径时,不包含子路径的cookie

Exipres: cookie 过期时间. 不设置则为session会话期,页面退出时cookie失效

HttpOnly: 设置为true时,只有http能读取,js只能读取未设置HttpOnly的cookie

Secure: 标记为Secure的cookie,只有https的请求可以携带

SameSite:限制第三方url(跨域)是否可以携带cookie
* strict: 仅允许发送同站点请求的cookie
* Lax允许部分第三方请求携带cookie 允许跨站get请求发送
* None 任意发送cookie 设置为None (需要同时设置为Secure,也就是说网站必须采用https)

9.cookie大小和数量限制

不同浏览器大小数量限制不同

只要遵守如下规则

  • 不超过300个cookie
  • 每个cookie不超过4k
  • 每个域名不超过20个cookie

每个域能设置的cookie总数也是有限制的 不同浏览器限制不同
最新版ie和Edge 每个域不超过50个cookie
最新版firefox 每个域不超过50个cookie
最新版Opera 每个域不超过180个cookie
Safari和Chrome 没有硬性限制

10. 查看浏览器是否打开Cookie功能

window.navigator.cookieEnabled

11. cookie的共享策略

domain和path 共同决定了cookie可以被哪写url访问
访问url时, 如果url的host和domian一致或者是domain的子域名,并且url的路径与path路径匹配
那么cookie可以被读取

12. 跨域请求(CORS)中的cookie

首先cookie的SameSite需要设置为None 其次对于将Access-Control-Allow-Credential 设置为true的
接口(表示允许发送cookie),需要我们在发送请求时,将widthCredentials 设为true

13. cookie的编码

cookie中 存在等于号,冒号,分号,空格,都好,斜杠等特殊字符

需要对其进行转码,一般会用
encodeURIComponent /decodeURIComponent 进行操作

为什么不使用escape和encodeURI进行编码

因为相比这两个api encodeURIComponent 可以将更多的字符进行转码 比如 ‘/‘

14. cookie与web安全

  1. cookie如何因对XSS漏洞

XSS漏洞的原理是,由于未对用户提交的表单数据或者url参数等数据做处理就显示在了页面上,导致用户提交的内容在页面上被html解析执行

常规方案

对特殊字符进行处理 如’<’ 和’>’ 等进行转义

cookie应对方案

对于用户利用script脚本来采集cookie信息,我们可以将重要的cookie设置为HttpOnly 来避免cookie被js采集

  1. cookie如何应对CSRF攻击

CSRF 中文名叫跨站请求伪造,原理是,用户登录了A网站,然后因为某些原因访问了B网站(比如跳转等),B网站直接发送一个A网站得请求进行一些危险操作
由于A网站处于登录状态,就发生了CSRF攻击(核心就是利用了cookie信息可以被跨站携带)

常规方案 采用验证码或token等

cookie的应对方案

由于CSRF攻击核心就是利用了cookie信息可以被跨站携带,那么我们可以对核心cookie的SameSite 设置为Strict 或Lax来避免

15. 不同的浏览器共享cookie吗

不同的浏览器互相不通信,彼此是独立cookie,因此,我们在一个浏览器上登录某个网站,换到另一个浏览器上的时候需要重新登陆

16. 哪些信息适合放到cookie中

cookie的增多无疑会加重网络请求的开销,而且每次请求都会将cookie完整的带上,因此对于那些’每次请求都必须携带的信息’(如身份信息\A/B分桶信息等),才适合放进cookie中,
其它类型的数据建议放进localstorage中
